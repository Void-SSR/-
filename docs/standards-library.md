# 标准库（Web 版）

最后更新：2026-03-19

这份标准库用于沉淀《拯救美少女》Web 版的长期协作标准、工程标准、美术标准与安全标准。
后续 Web 版任务优先复用这里的标准；如遇更优方案，在说明原因并获得确认后升级本库。

## A. 流程标准

### A-01 需求双确认流程
- 对需要实际执行的任务，先输出 `需求理解`
- 用户回复 `OK`
- 再输出 `本次任务对应的世界权威高标准执行方式`
- 用户再次回复 `OK`
- 之后才进入任务面板、执行和跟踪

### A-02 任务面板开启时机
- 当一轮需求超过 `2` 个时，进入执行前必须开启原生任务面板
- 任务面板在第二次 `OK` 之后开启，不提前开启

### A-03 普通问答例外
- 纯提问、解释、判断、咨询，不强制走双确认流程
- 这类对话按默认协作方式直接回答

### A-04 网址提供规则
- 只有当对应改动已经确认成功时，才提供对应网址
- 改动 `Web 版日志板` 时，只提供 `Web 版日志板网址`
- 改动 `Web 版游戏` 时，只提供 `Web 版游戏网址`
- 如果两边都改了，再分别提供两个网址
- 在需求理解、标准对齐或尚未成功的阶段，不提前提供网址

### A-05 错误库持续改进规则
- Web 版必须维护独立错误库，记录未达成目标、卡点、根因、解决方式与沉淀规范
- 错误库不是抱怨清单，而是用于避免重复犯错的正式经验库
- 后续每次出现新的卡点或未达成目标，都应进入错误库并持续迭代

## B. 工程标准

### B-01 工具与方法选择标准
- 优先官方文档、官方发行渠道、主流长期维护工具
- 优先兼容、稳定、干净、可持续迭代的方案
- 避免冷门、投机、兼容性差或残留重的方式

### B-02 新工具引入标准
- 任何新工具第一次正式使用前，都必须先做安全验证
- 不以“能安装/能运行”作为安全通过标准
- 验证通过后才纳入正式工具链

### B-03 下一个版本标签启用标准
- 版本标签不是按修改次数递增
- 只有形成新的正式里程碑时，才启用下一个版本标签
- 新里程碑至少应满足：
  - 范围明确
  - 已完成必要验证
  - 适合作为稳定回滚点
  - 日志/看板已同步
  - 能被一句清楚的话命名
- 中间迭代可以长期停留在当前版本标签下，不必因为小改动频繁升级标签

## C. 产品与交互标准

### C-01 移动端分层标准
- 优先使用 `2/3 级页面分层`
- 尽量避免通过超长单页滑动完成主要操作
- 首页以摘要、主入口和状态概览为主，详细内容下沉到二级/三级页面

### C-02 商业化高质量标准
- Web 版每个板块都要按高质量、商业化标准推进
- 不能只让首页看起来像游戏，而其他板块仍然像 demo
- 至少覆盖：首页、章节、战斗、图鉴、商城、装备、结算、日志看板

## D. 美术标准

### D-01 游戏化风格标准
- UI 不能停留在程序原型状态
- 必须建立统一、明确、适合题材的游戏化视觉风格
- 必要时使用图片化按钮、底板、装饰和主视觉资源

### D-02 原画风标准
- 后续主视觉资源优先采用商业化原画路线
- 不再继续依赖简陋矢量占位图作为最终表达

### D-03 角色风格标准
- 美女角色按成熟、精致、性感、商业化角色原画标准设计
- UI 风格、角色原画和场景气质要统一

## E. 安全标准

### E-01 新工具安全验证标准
- 新工具在正式使用前，必须先做来源、治理、完整性、依赖与残留风险验证
- 优先使用世界权威高标准的验证方法
- 如果现有验证能力不够，先补安全验证工具链，再引入新工具

### E-02 安全验证工具链基线
当前正式基线：
- 系统原生：`spctl`、`codesign`、`pkgutil`、`shasum`、`xattr`、`otool`、`file`
- 供应链签名验证：`cosign`
- 文件系统/依赖扫描：`trivy`
- 版本与发布大文件支持：`git-lfs`
